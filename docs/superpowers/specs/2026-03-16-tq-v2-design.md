# tq v2 — Reimagined

Pure Python. Conversation-first. Karpathy-simple.

## Philosophy

Every top-level Telegram message spawns a Claude Code session in tmux.
Reply to continue. That's it.

## Two Rules

1. **Reply-to a tracked message** → route to that session
2. **Anything else** → new session

## Architecture

```
~/.tq/tq.db          ← all state (SQLite)
~/.tq/config.json    ← telegram bot token, chat id
~/.tq/hooks/         ← generated per-session hooks

tq/
  cli.py             ← entry point + subcommands
  daemon.py          ← telegram long-poll + tmux health
  session.py         ← tmux spawn/stop/route + hooks
  store.py           ← SQLite schema + queries
  telegram.py        ← Bot API send/receive
```

Five files. ~500 lines total. Zero external dependencies (Python stdlib only).

## Data Model

```sql
CREATE TABLE sessions (
    id TEXT PRIMARY KEY,
    prompt TEXT,
    cwd TEXT NOT NULL,
    status TEXT NOT NULL DEFAULT 'pending',
    tmux_session TEXT,
    queue TEXT,
    created_at TEXT NOT NULL,
    started_at TEXT,
    completed_at TEXT
);

CREATE TABLE messages (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    telegram_msg_id INTEGER UNIQUE,
    session_id TEXT NOT NULL REFERENCES sessions(id),
    direction TEXT NOT NULL,  -- 'in' or 'out'
    text TEXT,
    created_at TEXT NOT NULL
);
```

Two tables. No flat files. No registry.json. No scattered `.tq/` directories.

## CLI

```
tq daemon start|stop|status    # the brain
tq status                      # show all sessions
tq stop <id>                   # kill a session
tq run [file|prompt] [--cwd]   # queue file or one-shot
tq setup                       # configure telegram
```

That's the whole CLI. Five subcommands.

## Daemon

Single long-running Python process. Does two things:

1. **Telegram long-poll** — fetches messages, routes them
2. **Health loop** — checks tmux sessions every 30s, marks dead ones done

Routing logic (the whole thing):

```python
def route(msg):
    if msg.reply_to:
        session = store.lookup_message(msg.reply_to)
        if session:
            return send_to_session(session, msg)
    return spawn_new_session(msg)
```

## Session Lifecycle

**Spawn:**
1. Insert into `sessions` table (status=running)
2. Generate `~/.tq/hooks/<id>/on-stop.sh` and `settings.json`
3. `tmux new-session -d -s tq-<id>`
4. Send: `claude --settings <path> --dangerously-skip-permissions <prompt>`
5. Track the incoming telegram msg in `messages` table

**Route (existing session):**
1. `tmux load-buffer` + `paste-buffer` (safe for special chars)
2. Track the incoming msg in `messages` table

**Reply (Claude responds via /tq-reply):**
1. CLI writes outgoing msg to `messages` table
2. Sends via Telegram API (threaded to the incoming msg)

**Stop (session ends):**
1. on-stop.sh fires → `tq _mark-done <id>`
2. Updates `sessions.status = 'done'`, sets `completed_at`

**Health check:**
1. Every 30s, daemon checks `tmux has-session` for all running sessions
2. Dead sessions get marked done

## Queue Files (Optional)

Queue files are just a batch way to create sessions:

```yaml
cwd: ~/project
tasks:
  - Review yesterday's commits
  - Run the test suite
  - Add documentation
```

`tq run morning.yaml` → inserts N sessions into SQLite → spawns them.

Supports:
- `schedule: "0 9 * * *"` — cron scheduling
- `reset: daily` — auto-clear state
- `sequential: true` — one at a time
