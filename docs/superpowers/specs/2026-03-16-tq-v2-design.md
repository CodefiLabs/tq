# tq v2 — Reimagined

Pure Python. Conversation-first. Karpathy-simple.

## Philosophy

Every top-level Telegram message spawns a Claude Code session in tmux.
Reply to continue. That's it.

## Two Rules

1. **Reply-to a tracked message** → route to that session
2. **Anything else** → new session

## Architecture

```
~/.tq/tq.db          ← all state (SQLite)
~/.tq/config.json    ← telegram bot token, chat id
~/.tq/hooks/         ← generated per-session hooks

tq/
  cli.py             ← entry point + subcommands
  daemon.py          ← telegram long-poll + tmux health
  session.py         ← tmux spawn/stop/route + hooks
  store.py           ← SQLite schema + queries
  telegram.py        ← Bot API send/receive
```

Five files. ~500 lines total. Zero external dependencies (Python stdlib only).

## Data Model

```sql
CREATE TABLE sessions (
    id TEXT PRIMARY KEY,
    prompt TEXT,
    cwd TEXT NOT NULL,
    status TEXT NOT NULL DEFAULT 'pending',
    tmux_session TEXT,
    queue TEXT,
    created_at TEXT NOT NULL,
    started_at TEXT,
    completed_at TEXT
);

CREATE TABLE messages (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    telegram_msg_id INTEGER UNIQUE,
    session_id TEXT NOT NULL REFERENCES sessions(id),
    direction TEXT NOT NULL,  -- 'in' or 'out'
    text TEXT,
    created_at TEXT NOT NULL
);
```

Two tables. No flat files. No registry.json. No scattered `.tq/` directories.

## CLI

```
tq daemon start|stop|status    # the brain
tq status                      # show all sessions
tq stop <id>                   # kill a session
tq run [file|prompt] [--cwd]   # queue file or one-shot
tq setup                       # configure telegram
```

That's the whole CLI. Five subcommands.

## Daemon

Single long-running Python process. Does two things:

1. **Telegram long-poll** — fetches messages, routes them
2. **Health loop** — checks tmux sessions every 30s, marks dead ones done

Routing logic (the whole thing):

```python
def route(msg):
    if msg.reply_to:
        session = store.lookup_message(msg.reply_to)
        if session:
            return send_to_session(session, msg)
    return spawn_new_session(msg)
```

## Session Lifecycle

**Spawn:**
1. Insert into `sessions` table (status=running)
2. Generate `~/.tq/hooks/<id>/on-stop.sh` and `settings.json`
3. `tmux new-session -d -s tq-<id>`
4. Send: `claude --settings <path> --dangerously-skip-permissions <prompt>`
5. Track the incoming telegram msg in `messages` table

**Route (existing session):**
1. `tmux load-buffer` + `paste-buffer` (safe for special chars)
2. Track the incoming msg in `messages` table

**Reply (Claude responds via /tq-reply):**
1. CLI writes outgoing msg to `messages` table
2. Sends via Telegram API (threaded to the incoming msg)

**Stop (session ends):**
1. on-stop.sh fires → `tq _mark-done <id>`
2. Updates `sessions.status = 'done'`, sets `completed_at`

**Health check:**
1. Every 30s, daemon checks `tmux has-session` for all running sessions
2. Dead sessions get marked done

## Queue Files (Optional)

Queue files are just a batch way to create sessions:

```yaml
cwd: ~/project
tasks:
  - Review yesterday's commits
  - Run the test suite
  - Add documentation
```

`tq run morning.yaml` → inserts N sessions into SQLite → spawns them.

Supports:
- `schedule: "0 9 * * *"` — cron scheduling
- `reset: daily` — auto-clear state
- `sequential: true` — one at a time

tq v2 as a Claude Code plugin is minimal. The core is the Python package — the plugin
just teaches Claude how to interact with it.

### Structure

```

.claude/
  settings.json
  commands/
    tq-reply.md

skills/
  tq/
    SKILL.md
```

Four files. That's the entire plugin surface.

### plugin.json

```json
{
  "name": "tq",
  "version": "2.0.0",
  "description": "Claude Code sessions via Telegram + tmux",
  "author": { "name": "codefilabs" },
  "license": "MIT",
  "keywords": ["claude", "tmux", "telegram", "sessions"]

```

### settings.json

```json
{
  "permissions": {
    "deny": [
      "Read(**/.env*)",
      "Read(**/*.pem)",
      "Read(**/*.key)"
    ]
  }
}
```

Simpler than v1 — no need to deny `.tq/` or `.launch.py` since those don't exist anymore.

### /tq-reply (the only slash command)

```yaml
---
name: tq-reply
description: Send response back to Telegram user
allowed-tools: Bash(tq2)
argument-hint: (no arguments — auto-detects session)
---
```

Body: Read `$TQ_SESSION_ID` from env, compose a concise Telegram-friendly response,
run `tq2 reply "$TQ_SESSION_ID" "<response>"`. Done.

The session ID is injected as an env var when the tmux session spawns —
no marker files, no tmux name parsing.

Body: Brief overview of the 5 CLI commands + the daemon model.
Under 50 lines. No need to document 13 slash commands anymore.

## OpenClaw Plugin

tq v2 as an OpenClaw plugin is a different shape. OpenClaw already handles
multi-channel messaging (Telegram, Slack, Discord, etc.) — so tq doesn't
need its own Telegram integration. It just provides the "spawn Claude Code
sessions in tmux" capability as tools + a background service.

### Key Insight

In Claude Code plugin mode: tq handles everything (telegram + tmux).
In OpenClaw plugin mode: tq handles tmux only. OpenClaw handles messaging.

This means the OpenClaw plugin is a **subset** of the Python package — it
imports `session.py` and `store.py` but NOT `telegram.py` or `daemon.py`.

### Structure

```
openclaw.plugin.json
package.json
tsconfig.json
src/
  index.ts          # register(api) — tools + service
  tq-bridge.ts      # Calls tq Python via subprocess
```

Two source files. The TypeScript is just a thin bridge to the Python core.

### openclaw.plugin.json

```json
{
  "id": "tq",
  "name": "tq — Claude Code Sessions",
  "description": "Spawn and manage Claude Code sessions in tmux",
  "kind": "tools",
  "version": "2.0.0",
  "configSchema": {
    "type": "object",
    "properties": {
      "defaultCwd": {
        "type": "string",
        "default": "~",
        "description": "Default working directory for new sessions"
      },
      "tqPath": {
        "type": "string",
        "default": "tq2",
        "description": "Path to tq2 CLI binary"
      }
    }
  }
}
```

No Telegram config — OpenClaw owns the messaging channel.

### Tools (3)

| Tool | Description |
|------|-------------|
| `tq_run` | Spawn a Claude Code session with a prompt and cwd |
| `tq_status` | List all sessions with status |
| `tq_stop` | Stop a running session |

These map 1:1 to `tq2 run`, `tq2 status`, `tq2 stop`.

### Service (1)

A background health-check service that runs every 30s, same as the daemon's
health loop but without Telegram polling (OpenClaw delivers messages via its
own channel system).

```typescript
api.registerService({
  name: "tq-health",
  interval: 30_000,
  execute: async () => {
    // Check tmux sessions, mark dead ones done
    execSync("tq2 _health-check");
  }
});
```

### Hook (1)

```typescript
api.on("before_agent_start", (ctx) => {
  // Inject running session context so the agent knows what's active
  const status = execSync("tq2 status --json").toString();
  return { prependContext: `Active tq sessions:\n${status}` };
});
```

### Message Flow (OpenClaw mode)

```
User sends message via any channel (Telegram/Slack/Discord/web)
    → OpenClaw receives it
    → Agent decides to use tq_run tool
    → tq spawns Claude Code session in tmux
    → Session completes → on-stop hook fires
    → tq_status shows "done"
    → Agent reports back via OpenClaw's channel
```

No direct Telegram integration. OpenClaw is the messaging layer.
tq is just the "Claude Code in tmux" engine.

### What tq provides that OpenClaw doesn't

OpenClaw can run agents, but it runs them in-process. tq provides:
- **tmux isolation** — each session is a separate tmux process
- **Claude Code interactive** — full `claude` CLI with hooks, permissions, chrome
- **Persistent sessions** — survive OpenClaw restarts
- **Queue files** — batch session creation from YAML

### Installation

```bash
# Install tq Python package
pip install tq  # or just symlink tq2 to PATH

# Install OpenClaw plugin
openclaw plugins install /path/to/tq/openclaw-plugin
```
