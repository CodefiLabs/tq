# AI Context — What Makes tq Tricky

This document captures the non-obvious behaviors and design decisions in `tq` that are likely
to trip up AI assistants. Read this before making any changes.

## The macOS `sed -i ''` Issue (Will Trip You Up)

The most common AI mistake in this codebase. BSD sed (macOS) requires an empty string argument
after `-i`; GNU sed (Linux) does not:

```bash
# CORRECT — macOS
sed -i '' 's/^status=running/status=done/' "$STATE_FILE"

# WRONG — GNU Linux style; on macOS this treats '' as a backup extension
sed -i 's/^status=running/status=done/' "$STATE_FILE"
```

This pattern appears in three places:
- `scripts/tq` (dead session detection in bash loop)
- `scripts/tq-status` (stale session reaping)
- `~/.tq/sessions/<hash>/hooks/on-stop.sh` (generated by the Python parser)

Do not change any of these to the GNU form.

## Two State Dirs That Look Similar But Are Different

```
<queue-dir>/.tq/<queue-basename>/<hash>      ← task state (status, session name, timestamps)
~/.tq/sessions/<hash>/settings.json          ← Claude session configuration (Stop hook path)
```

Both are keyed by the same 8-char SHA-256 hash, but they serve different purposes and are
read/written by different actors. The task state dir is adjacent to the queue YAML file.
The session dir lives in the user's home directory and is read by the `claude` CLI itself.

Common mistake: trying to find Claude settings in the queue dir, or writing state into `~/.tq/`.

## Embedded Python via Heredoc — Do Not Refactor

The Python YAML parser in `scripts/tq` is written to a temp file via heredoc:

```bash
PARSE_SCRIPT=$(mktemp /tmp/tq-parse-XXXXXX.py)
trap 'rm -f "$PARSE_SCRIPT"' EXIT
cat > "$PARSE_SCRIPT" <<'PYEOF'
import sys, os, hashlib, re, json, stat
# ... 170 lines of Python ...
PYEOF
python3 "$PARSE_SCRIPT" "$QUEUE_FILE" "$STATE_DIR"
```

This pattern exists for a specific reason: bash 3.2 has a bug where single quotes inside
`<<'PYEOF'` inside `$()` are incorrectly scanned by the quote counter. The temp file approach
avoids this entirely. Do not refactor this to `python3 -c "..."` or subprocess calls.

## `os.execvp()` in Launchers Is Intentional Process Replacement

Every generated `<hash>.launch.py` ends with `os.execvp(...)`:

```python
os.execvp('claude', ['claude', '--settings', settings_file, '--dangerously-skip-permissions', '--chrome', prompt])
```

`execvp` **replaces** the Python process with `claude` — the Python interpreter is gone after
this call. This is intentional: it means the tmux window shows the `claude` process directly,
not a Python wrapper. Do not change this to `subprocess.run()` or `subprocess.Popen()`.

## SHA-256[:8] Is the Stable Task Identity — Don't Touch It

```python
h = hashlib.sha256(prompt.encode()).hexdigest()[:8]
```

This hash is computed at queue-run time and used as the filename for state files, prompt files,
and launcher scripts. All three scripts (`tq`, `tq-status`, `on-stop.sh`) reference state
by this hash. **Changing the hashing logic orphans all existing state files** — tq will
re-run every task that was previously marked done.

## `--dangerously-skip-permissions` Is Required

Every spawned claude session receives this flag. It suppresses interactive permission prompts
that would block unattended execution. This is by design. Do not remove it.

## Chrome Profile 5 Is Hardcoded

In the generated launcher (`scripts/tq` line ~194):

```python
subprocess.Popen(["open", "-a", "Google Chrome", "--args", "--profile-directory=Profile 5"])
```

This opens Chrome with Profile 5 (`halbotkirchner@gmail.com`) before Claude connects.
It is not configurable per queue or per task. This is a known limitation.
If you need to change the profile, edit `scripts/tq` directly — search for `Profile 5`.

## Domain Terminology

| Term | Meaning |
|------|---------|
| queue | A YAML file defining a list of tasks (`~/.claude/queues/<name>.yaml`) |
| task | A single `prompt:` entry in the queue YAML |
| hash | `sha256(prompt_text)[:8]` — the stable task identity across runs |
| state file | `<queue-dir>/.tq/<name>/<hash>` — key=value store: status, session, started |
| launcher | `<hash>.launch.py` — generated Python script that `execvp`s into `claude` |
| session | A tmux window running a `claude` process for one task |
| Stop hook | `on-stop.sh` — shell script called by `claude` on exit; marks task done |
| pending | No state file exists yet (or file was deleted to reset) |
| running | State file has `status=running` and tmux session is live |
| done | State file has `status=done` (Stop hook fired or dead session detected) |

## YAML Parser Quirks

The Python parser in `scripts/tq` is hand-rolled (no YAML library). It handles:
- Inline: `- prompt: fix the login bug`
- Block literal: `- prompt: |` followed by indented lines
- Block folded: `- prompt: >` (newlines become spaces)
- Quoted inline: `- prompt: "fix the 'login' bug"`

It does **not** use PyYAML or any library — if you're adding YAML features, continue the
regex line-scanner pattern. Do not add `import yaml` or install any Python packages.

## Cron Context Requires PATH Export

`tq` explicitly sets PATH at the top:

```bash
export PATH="/opt/homebrew/bin:/usr/local/bin:$PATH"
```

This is required because cron runs with a minimal PATH that may not include Homebrew binaries
(`tmux`, `python3`, `claude`). Do not remove this line.
